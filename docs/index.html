StatelessWidget vs StatefulWidget

Comprendi le differenze fondamentali tra i due tipi principali di widget

Confronto dettagliato tra StatelessWidget e StatefulWidget

StatelessWidget

I StatelessWidget sono widget immutabili che non possono cambiare il loro stato una volta creati.

Caratteristiche:

  • Immutabili
  • Non dipendono da dati dinamici
  • Pi√π performanti
  • Si ricostruiscono solo quando il parent cambia

Esempio di codice:

class MyWidget extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Text('Ciao Flutter!');
  }
}

StatefulWidget

I StatefulWidget sono widget che possono cambiare il loro stato durante il runtime.

Caratteristiche:

  • Mutabili
  • Gestiscono stato interno
  • Possono ricostruire l'UI
  • Utilizzano setState()

Esempio di codice:

class MyWidget extends StatefulWidget {
  @override
  _MyWidgetState createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget> {
  int counter = 0;
  
  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        Text('Contatore: $counter'),
        ElevatedButton(
          onPressed: () {
            setState(() {
              counter++;
            });
          },
          child: Text('Incrementa'),
        ),
      ],
    );
  }
}

Tips & Best Practices

Consigli avanzati per sviluppare con Flutter

Performance

  • Usa const constructor quando possibile per widget immutabili
  • ListView.builder per liste grandi invece di ListView con children
  • Evita di ricostruire widget pesanti, usa chiavi per l'ottimizzazione
  • Separa widget in componenti riutilizzabili

Layout

  • Row e Column per layout lineari, Stack per sovrapposizioni
  • Expanded e Flexible per distribuzione spazio in Row/Column
  • SingleChildScrollView per contenuto che pu√≤ traboccare
  • MediaQuery per layout responsive

Gestione Stato

  • Usa StatelessWidget quando possibile per performance
  • setState() per stato locale semplice
  • Provider, Bloc o Riverpod per gestione stato complessa
  • Evita setState() in build() per evitare loop infiniti
